## Step 3 — NCCL_DEBUG=INFO live capture (TP=2 → post-fix evidence of NET/IB vs NET/Socket)

### 3.1 First attempt — FAILED — 2026-04-18 13:40 UTC

Configured Spark 1 as head (`distributed_mode=head, cluster_role=master,
peer_ips=[192.168.0.219], model=Qwen/Qwen2.5-14B-Instruct`) and Spark 2 as
member worker. Restarted Spark 2 then Spark 1 to trigger distributed launch.

Monitor fired with this error:
```
Error: ibdev2netdev not found. Cannot auto-detect interfaces.
```

**Root cause.** AINode's new `_detect_ib_hca()` in commit 1 (v0.4.9) shells
out to `ibdev2netdev` — a tool that ships with MOFED / infiniband-diags on
the HOST, not in the ainode container or eugr base image. Detection returns
`None` → `_write_eugr_env()` writes `IB_IF=` (empty). Empty `IB_IF` triggers
eugr's `autodiscover.sh` detect-interfaces path, which also depends on
`ibdev2netdev` and fails with the same error. Launch aborts.

Concrete evidence captured:
```

--- .env that AINode wrote on this failed launch (note empty IB_IF) ---

ETH_IF=enP2p1s0f1np1
IB_IF=
CONTAINER_NCCL_SOCKET_IFNAME=enP2p1s0f1np1
CONTAINER_NCCL_IB_DISABLE=0
...
CONTAINER_AINODE_CLUSTER_SUBNET=192.168.0.0/24

--- distributed.log showed eugr autodiscover bailing on the same tool ---
Using launch script: /opt/spark-vllm-docker/examples/ainode-distributed.sh
Error: ibdev2netdev not found. Cannot auto-detect interfaces.
(restarts repeated; systemd Restart=on-failure kept re-triggering)
```

**Verification-gate caught the bug.** The plan said stop on failure, report,
propose a fix, get approval before re-cycling — exactly what happened. Not
merging to main until this is resolved.

**Fix in progress:** commit 7 replaces `ibdev2netdev` with direct reads of
`/sys/class/infiniband/` in both AINode's Python detector and the shim.
Sysfs is kernel-provided and always available in any Linux container. Port
state parse uses the leading integer and matches against the `ib_port_state`
kernel enum (`include/rdma/ib_verbs.h`) — accepting 4 (ACTIVE) and 5
(ACTIVE_DEFER), rejecting everything else. Format-stable across kernel
versions where the textual label may vary.

### 4.3 Interpretation

Both HCAs pass RoCE v2 traffic at 104–109 Gbps average between Spark 1
and Spark 2 via the MikroTik CRS812 switch. No hangs, no errors, clean
5000-iteration completion. The fabric is functional at the wire level.

**This rules out network-layer causes** for the NCCL hang: the switch is
forwarding RoCE v2 (UDP 4791) correctly, PFC/ECN aren't dropping frames
at these rates, QSFP links are healthy, GID resolution works, and
reliable-connection QPs complete full bidirectional traffic without
retransmits or drops.

**Tomorrow's investigation direction: software layer.**

Three candidates to pursue, in order of my current-evidence confidence:

1. **`nvidia_peermem` / GPU Direct RDMA.** Not loaded on any of the 4
   Sparks (`lsmod | grep nvidia_peermem` returns empty). NCCL logs from
   an earlier attempt explicitly said `NET/IB : GPU Direct RDMA Disabled
   for HCA 0 'mlx5_0'`. Without GDR, NCCL still establishes IB connections
   and `ib_write_bw` works fine (it doesn't use CUDA memory regions), but
   NCCL's GPU-memory-registration code path may hang differently or not
   complete the rendezvous at all. Load `nvidia_peermem` (available in
   MOFED userspace; may need kernel-module rebuild against
   `6.17.0-1014-nvidia`) and retry.
2. **eugr's `dgxspark-3node-ring` patched NCCL.** The patch is in the
   base image (NCCL version string reads `2.29.7+cuda13.2` with git
   tag `dgxspark-3node-ring fab1850`). Patch is optimized for a 3-node
   direct-connect triangle, not a 2- or 4-node switch topology. Possible
   the patch makes assumptions that don't hold on 2 nodes. Test by
   rebuilding the base image against vanilla NCCL and re-running TP=2.
3. **NCCL bootstrap subnet.** NCCL uses a TCP bootstrap phase before IB
   rendezvous. The bootstrap address is picked from NCCL_SOCKET_IFNAME,
   currently `enP2p1s0f1np1` (= `mlx5_3` = 192.168.0.63/65). Bootstrap
   goes over the same switch so should work, but worth confirming via
   `NCCL_DEBUG_SUBSYS=INIT,NET` log analysis on a run that reaches this
   phase.

## Lessons learned

### Class of bug: host-tested, container-runs

The shim called two tools from the host's userspace:
- `ibdev2netdev` (commits 1 & 7 delta) — provided by MOFED /
  `infiniband-diags`.
- `ip` from `iproute2` (commit 8 delta) — provided by the standard
  Ubuntu `iproute2` package.

Neither is present in the eugr vllm-node base image. Both bugs were
caught at verification time, not design time.

**Root cause:** During shim design we verified commands worked by running
them on the host. The shim executes inside a container with a different
package set.

**Defense for v0.4.10 and beyond:**

- **Prefer kernel ABIs over userspace tools.** `/sys/class/infiniband/`
  (commit 7) and `ioctl SIOCGIFADDR` (commit 8) are always present in any
  Linux container because they come from the kernel itself. `ibdev2netdev`
  and `ip` are userspace packages that vary by distribution and may be
  absent.
- **Add a shim self-test stage to CI** that runs inside the target
  container image and fails the build if any `command -v <tool>` check
  finds a missing dependency.
- **Reviewer checklist for any entrypoint-shim change:** for every
  external command called, is it in the target image? Tested inside the
  container, not on the host.

Both instances today were caught at verification, not in production —
the discipline of running the shim end-to-end against the real container
surface (not a host mock) is what exposed them. Continuing that discipline
will catch the next one earlier, ideally at design time before shipping.
