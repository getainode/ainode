**Tomorrow's investigation direction: software layer.**

Three candidates to pursue, in order of my current-evidence confidence:

1. **`nvidia_peermem` / GPU Direct RDMA.** Not loaded on any of the 4
   Sparks (`lsmod | grep nvidia_peermem` returns empty). NCCL logs from

2. **eugr's `dgxspark-3node-ring` patched NCCL.** The patch is in the

## Lessons learned

### Class of bug: host-tested, container-runs

The shim called two tools from the host's userspace:
- `ibdev2netdev` (commits 1 & 7 delta) ŌĆö provided by MOFED /
  `infiniband-diags`.
- `ip` from `iproute2` (commit 8 delta) ŌĆö provided by the standard
  Ubuntu `iproute2` package.

Neither is present in the eugr vllm-node base image. Both bugs were
caught at verification time, not design time.

**Root cause:** During shim design we verified commands worked by running
them on the host. The shim executes inside a container with a different
package set.

**Defense for v0.4.10 and beyond:**

- **Prefer kernel ABIs over userspace tools.** `/sys/class/infiniband/`
  (commit 7) and `ioctl SIOCGIFADDR` (commit 8) are always present in any
  Linux container because they come from the kernel itself. `ibdev2netdev`
  and `ip` are userspace packages that vary by distribution and may be
  absent.
- **Add a shim self-test stage to CI** that runs inside the target
  container image and fails the build if any `command -v <tool>` check
  finds a missing dependency.
- **Reviewer checklist for any entrypoint-shim change:** for every
  external command called, is it in the target image? Tested inside the
  container, not on the host.

Both instances today were caught at verification, not in production ŌĆö
the discipline of running the shim end-to-end against the real container
surface (not a host mock) is what exposed them. Continuing that discipline
will catch the next one earlier, ideally at design time before shipping.

### Remaining suspects, not yet investigated

1. **`nvidia_peermem` not loaded.** NCCL says `GPU Direct RDMA Disabled
   for HCA 0/1`. GDR requires the `nvidia_peermem` kernel module.
   Without it NCCL falls back to a host-staging path. On switched RoCE
   with vanilla NCCL 2.28.3, the fallback path may be broken or
   untested. Prior user instruction: do not load `nvidia_peermem`
   without explicit approval.

2. **`mlx5dv_reg_dmabuf_mr` symbol missing.** The container's libmlx5
   (rdma-core userland) is older than the `MLX5_1.25` ABI that NCCL
   2.28.3 tries to use for DMA-BUF memory registration. NCCL logs this
   as a warning but continues ŌĆö the question is whether the fallback
   path actually works end-to-end or deadlocks mid-connection-setup.

3. **Dual-HCA ring.** Shim selected two HCAs per node (`mlx5_1` and
   `mlx5_3`). NCCL builds channels that straddle both. If the peer
   reachability of `mlx5_1Ōåömlx5_1` differs from `mlx5_3Ōåömlx5_3`
   (different VLAN, different switch port, different MTU), NCCL
   may deadlock waiting for a channel that can't complete. `ib_write_bw`
   (Step 4) tested only one port-pair, not both, so this is untested.

### Escalation

This is the third NCCL-init failure in v0.4.9. Per the session rule, no
further intervention is made. Options presented to user:

**Option A ŌĆö load `nvidia_peermem` on all 4 nodes.** Lowest-risk
next step IF user approves. Confirms/rules out hypothesis (1). Reversible.

**Option B ŌĆö force single-HCA per node.** Tests hypothesis (3) without
kernel module changes. Would require either extending the shim to
select the first matching HCA only, or setting `NCCL_IB_HCA=mlx5_1`
explicitly via env override, and rerun.

**Option C ŌĆö rebuild with newer rdma-core in container.** Tests
hypothesis (2). Highest-cost ŌĆö requires adding rdma-core bump to the
eugr Dockerfile patch in `build-base-image.sh`. Also the warning
may be cosmetic (NCCL continues past the dlvsym failure), so this is
the least-likely-to-help option.

**Option D ŌĆö run NCCL's own `nccl-tests` `all_reduce_perf` binary
directly in the vllm_node containers, bypassing vLLM/Ray.** This isolates
whether the hang is in NCCL proper or in the vLLM init path. Evidence-positive
regardless of outcome.
