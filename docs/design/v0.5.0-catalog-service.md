# v0.5.0 — AINode Catalog Service

**Status:** Draft · **Target release:** v0.5.0 · **Owner:** Jason Brashear · **Last updated:** 2026-04-17

A hosted model catalog service at `ainode.dev` that replaces the in-container `CatalogAggregator`. Serves as the single source of truth for model metadata consumed by (a) every deployed AINode node and (b) a public browse UI at `ainode.dev/models`.

---

## 1. Motivation

The current catalog is a live HuggingFace scrape performed inside every AINode container, with a 24h on-disk cache. Three problems:

1. **Coverage gaps.** The current sweep (`filter="text-generation", sort="downloads", limit=50`) misses models that matter — e.g. `deepseek-ai/DeepSeek-V3-0324` isn't in the top-100 download ranking. Users can't discover them in the app.
2. **No metadata enrichment.** Size estimation falls back to regex-parsing the model name. DeepSeek-V3.2 shows as "~0 GB". vLLM compatibility isn't checked. GB10-specific notes (AWQ auto-upgrade trap, Marlin kernels missing for sm_12.1) aren't captured anywhere.
3. **No cluster-awareness.** The "Fits GPU" badge uses single-node VRAM only. A user with 4×128 GB (487 GB aggregate) sees "Too large" for anything >128 GB, even though their cluster handles it trivially.

Shipping a better catalog inside the container makes this worse (larger image, staler data, multiple upgrade paths to manage). The right move is to decouple: catalog lives server-side, nodes fetch on startup with ETag caching.

---

## 2. Non-goals

- **Not hosting model weights.** HF serves weights. We serve pointers + metadata. Zero bandwidth cost to AINode.
- **Not a training-data catalog.** Models only. Datasets are out of scope for v0.5.0.
- **Not a replacement for HF search.** We curate; HF is still the primary discovery surface for long-tail models. Our "Paste HF repo" input remains the escape hatch.

---

## 3. Core architectural rule

> **Static read path = R2. Dynamic queries + mutations = Railway.**

This single sentence governs all routing decisions. If a future-engineer is unsure where something goes: if it's idempotent and read-heavy and cacheable, it goes on R2. If it needs a `WHERE`, a mutation, or auth, it goes on Railway.

## 5. Stack decisions

| Component | Choice | Rationale |
|-----------|--------|-----------|
| Hot JSON feed | **Cloudflare R2** | Zero egress, edge CDN, ETag-native. Unbounded read scale at ~$0 cost. |
| Postgres | **Railway Postgres plugin** | Already on Railway. No additional vendor. |
| Job queue | **BullMQ on Redis (Railway plugin)** | Durable queue, retries, observable. `node-cron` alone is fine for triggers but BullMQ handles the `enrich-metadata` fan-out cleanly. |
| Worker runtime | **Node.js service, separate Railway service** | Long-running, 24/7 uptime, no serverless time limits. Same repo as Next.js for code sharing via `packages/catalog-core`. |
| Public + admin UI | **Next.js on Railway** (existing `ainode.dev` service) | Already deployed. Single auth boundary. |
| Admin auth | **WorkOS AuthKit** | B2B-first (SSO, SCIM, audit logs). Free to 1M MAU. Target audience is MSPs, not consumers. |
| HF publishing | **`ainode-community` HF org** | Namespace reserved for future GB10-optimized quant republishing. Not used in v0.5.0 itself — reserved now to prevent squatting. |
| Node → catalog | **HTTP + ETag to `catalog.ainode.dev/catalog.json`** | Standard, cacheable, works behind proxies, dead-simple client code. |

### Why not…

- **Cloudflare Workers + D1 + KV.** Would force moving ainode.dev off Railway or splitting origin between two providers. Keeping everything on Railway reduces operational surface. R2 is the only CF piece we adopt, chosen specifically because it has no good Railway equivalent (zero-egress global CDN).
- **Clerk.** Consumer-oriented. WorkOS fits B2B better for the admin panel.
- **Static JSON in the `ainode` repo.** Every catalog change = PR + merge + redeploy. Doesn't scale past a handful of manual curations.
- **GitHub Pages for the JSON feed.** No CORS control, no signed URLs, no access logs.

## 7. Public API surface

All under `ainode.dev`. No auth required except where noted.

| Endpoint | Purpose | Hosted where |
|----------|---------|--------------|
| `GET catalog.ainode.dev/catalog.json` | Full feed, ETag-cached, schema-versioned | **R2** |
| `GET catalog.ainode.dev/vendors/{vendor}.json` | Per-vendor pre-rendered feed | **R2** |
| `GET catalog.ainode.dev/models/{slug}.json` | Per-model detail blob | **R2** |
| `GET catalog.ainode.dev/schema-version` | Plain text, e.g. `1` — client can pre-check | **R2** |
| `GET ainode.dev/api/catalog/search?q=&vendor=&min_size=` | Full-text + filter search | **Railway** |
| `GET ainode.dev/api/catalog/trending` | Live trending by download velocity | **Railway** |
| `GET ainode.dev/api/catalog/cluster-fit?vram_gb=&nodes=` | Returns models that fit given cluster | **Railway** |
| `GET ainode.dev/api/admin/catalog/pending` | WorkOS: list entries awaiting review | **Railway** |
| `POST ainode.dev/api/admin/catalog/{repo}/approve` | WorkOS: promote pending → active | **Railway** |
| `POST ainode.dev/api/admin/catalog/{repo}/deprecate` | WorkOS: flag deprecated | **Railway** |
| `PUT ainode.dev/api/admin/catalog/{repo}/gb10-notes` | WorkOS: edit compat metadata | **Railway** |
| `POST ainode.dev/api/admin/catalog/{repo}/staff-pick` | WorkOS: promote to staff pick | **Railway** |

### Schema versioning

`catalog.json` has a top-level `"schema_version": 1` field. AINode node client validates on fetch:

- Matches known version → use it.
- Unknown version → log warning, fall back to baked-in snapshot shipped with the container.
- Next major catalog schema change becomes `"schema_version": 2`. Old nodes keep working against `catalog-v1.json` for a deprecation window (6 months).

We maintain both files on R2 during the window: `catalog.json` → latest, `catalog-v1.json` → frozen.

## 8. Worker jobs

Running in a separate Railway service (`Dockerfile.worker`). Schedules managed via BullMQ repeatable jobs.

| Job | Schedule | Purpose |
|-----|----------|---------|
| `refresh-vendor-sweeps` | hourly | For each tracked vendor, `HfApi.list_models(author=v, sort="downloads", limit=30)`. Upserts into `models`. Refreshes download counts and `last_modified`. |
| `watch-new-releases` | every 15 min | For each tracked vendor, `sort="createdAt", limit=5`. Any repo missing from `models` → insert with `status='pending_review'`, enqueue enrichment. |
| `enrich-metadata` | queue-driven | For a given `hf_repo`: `HfApi.repo_info(repo, files_metadata=True)` → compute real `size_gb`. Pull `config.json` → extract `architecture`. Look up against vLLM registry → set `vllm_supported`. |
| `publish-to-r2` | after any write to `models` | Re-renders `catalog.json`, per-vendor JSONs, per-model JSONs. PUTs to R2 with `Cache-Control: public, max-age=60, s-maxage=300`. |
| `dispatch-gb10-test` | on new model approval | Triggers a GitHub Action on the self-hosted Spark runner to do a vLLM engine init dry-run. Result lands in `model_gb10_notes.known_issues`. |
| `archive-stale` | weekly | Models with `hf_last_modified > 365 days` and `hf_downloads < 10k` → flip to `status='archived'`. Hidden from default catalog but queryable. |

### Tracked vendors (tier 1)

`meta-llama`, `Qwen`, `deepseek-ai`, `mistralai`, `google`, `microsoft`, `nvidia`, `openai` (gpt-oss), `unsloth`, `lmstudio-community`

Expandable via admin UI. New vendors flow to `watch-new-releases` on their next run.

## 9. HF API rate-limit strategy

Unauthenticated HF = 1000 req/hour. Per-vendor sweeps across 10 vendors × hourly × 30 repos metadata = ~300 req/hour baseline, plus bursty enrichment. We'd hit the ceiling.

**Mitigations:**
1. Worker authenticates with `HF_TOKEN` (stored in Railway secrets). Authed users get dramatically higher limits.
2. Respect `Retry-After` header — BullMQ backoff with jitter.
3. Only call `repo_info(files_metadata=True)` when: a) model is new, b) `hf_last_modified` changed, or c) forced manually from admin UI. Never on every sweep.
4. Batch model listing uses cursor pagination, not repeated full-list calls.

## 10. Degraded behavior

The node client (`ainode/models/hosted_catalog.py`, new module replacing the aggregator) must handle:

| Failure | Behavior |
|---------|----------|
| `catalog.ainode.dev` DNS failure | Use baked-in snapshot (`ainode/models/catalog-snapshot.json`, refreshed at container build time). Log warning, retry at next startup. |
| 5xx from R2 | Serve from last successfully-cached response (`~/.ainode/catalog-cache.json`), TTL 7 days for stale-while-error. |
| Schema version mismatch | Fall back to baked-in snapshot, log loud warning, emit OpenTelemetry metric. |
| Partial JSON / parse error | Fall back to baked-in snapshot, mark cache invalid. |
| No network at all (air-gapped MSP deployment) | Baked-in snapshot is always sufficient for install. `ainode config --catalog-offline` flag pins to snapshot and suppresses network attempts. |

Snapshot is refreshed at every AINode CI image build — `scripts/Dockerfile.ainode` has a build step that `curl`s the current `catalog.json` into the image. Never older than the container.

## 11. R2 publish atomicity

Postgres is the source of truth. R2 is a derived view. They CAN drift if we write to Postgres but the publish fails.

**Strategy:** publish-before-commit is not viable (no transactional R2). Instead:

1. Worker writes model changes to Postgres inside a transaction.
2. On commit success, enqueues a `publish-to-r2` job.
3. The publish job reads current Postgres state → renders all JSON files → writes to R2.
4. If publish fails, job retries with BullMQ exponential backoff. Ten failures → dead-letter queue + alert.
5. A daily `reconcile` job: reads Postgres, re-renders, diffs against R2, republishes if drift detected.

**Upshot:** R2 is eventually consistent with Postgres, typically <60 seconds. Nodes poll at 1-hour cadence, so drift is invisible in normal operation. The reconcile job closes the worst-case gap.

## 12. Observability

- Worker emits structured logs to Railway stdout (captured automatically).
- Each job run writes a row to `refresh_history` with status + stats.
- `ainode.dev/api/health` returns `{worker_last_success, postgres: ok, redis: ok, r2: ok}` — pollable by uptime monitor.
- Railway built-in metrics cover CPU/memory/deploys. No additional APM needed at v0.5.0 scale.
- Admin UI surfaces the `refresh_history` table as a timeline view on `/admin/catalog/jobs`.

## 13. CORS

R2 bucket `catalog.ainode.dev` needs explicit CORS config:

```json
{
  "AllowedOrigins": ["https://ainode.dev", "https://docs.ainode.dev", "http://localhost:8765"],
  "AllowedMethods": ["GET", "HEAD"],
  "AllowedHeaders": ["If-None-Match", "If-Modified-Since"],
  "ExposeHeaders": ["ETag", "Last-Modified"],
  "MaxAgeSeconds": 3600
}
```

The `http://localhost:8765` allowance enables the public `ainode.dev/models` page to detect a running local AINode and offer a "Download on my AINode" deep link.

## 14. Authn + authz

**Authentication:** WorkOS AuthKit — email + SSO. One WorkOS organization (`ainode`) with multiple users.

**Authorization (v0.5.0, keep simple):**

| Role | Can |
|------|-----|
| `viewer` | Read admin pages, see pending queue, no mutations |
| `editor` | Approve pending entries, edit gb10_notes, promote staff picks |
| `admin` | All of editor + manage admin_users, trigger manual refreshes, deprecate models |

Role stored in `admin_users.role`. Middleware checks on every `/api/admin/*` route.

**Audit trail:** every mutation writes to `audit_log`. Immutable append-only — no delete/update on that table.

**Future (v0.5.1+):** community contribution flow (public form → admin review queue → approve/reject). Out of scope for v0.5.0.

## 15. Migration from current aggregator

Nodes upgrading from ≤0.4.8 to ≥0.5.0 get a one-shot migration on first boot:

1. Old code paths (`CatalogAggregator.fetch()`, `_fetch_huggingface_popular()`, etc.) are removed from the node.
2. New `hosted_catalog.py` fetches `catalog.ainode.dev/catalog.json` on startup.
3. If unreachable, falls back to baked-in snapshot.
4. Old `~/.ainode/catalog-cache.json` is deleted on first successful fetch (schema is different).
5. No DB migrations on the node — all node state stays in `~/.ainode/`.

Cluster-wide: `ainode update` via the master picks up the new image, cluster stays functional throughout (discovery + inference are independent of catalog).

## 17. Open questions

- [ ] **HF org name.** `ainode-community` vs. `ainode` vs. `getainode`. Leaning `ainode-community` to mirror LM Studio's pattern. **Decision needed before v0.5.0-beta.**
- [ ] **Search engine.** Postgres full-text is probably sufficient at the catalog's scale (low thousands of entries). If latency >100ms, upgrade to Meilisearch on Railway. **Revisit if search p95 >100ms at beta.**
- [ ] **Public submission flow** (community proposes a model → admin reviews). Worth it in v0.5.0 or defer to v0.5.1? **Deferring to v0.5.1 unless community demand surfaces earlier.**
- [ ] **Vendor logo source of truth.** Store in R2 `assets.ainode.dev/vendors/*.svg` — manual upload via admin UI, or scrape from HF avatar URL? **Start with manual; automate if it becomes a chore.**
- [ ] **Caching headers on `catalog.json`.** `max-age=60, s-maxage=300` means nodes re-fetch hourly at worst. Is hourly freshness enough for pending-release watchers? **Probably yes; nodes don't care about <24h-old entries.**
- [ ] **Rate limit on public search API.** Do we need per-IP rate limiting on `/api/catalog/search` to prevent scraping? **Add Cloudflare in front of Railway if scraping becomes an issue.**

- **2026-04-17:** Chose Railway + R2 + WorkOS over full Cloudflare stack. Rationale: ainode.dev already on Railway, operational familiarity outweighs CF's edge compute advantages at current scale. R2 is the only CF component adopted (no Railway equivalent for zero-egress CDN).
